# 顶级化开发执行规范

## 1. 文档定位

本文档是 `stock_ultimate_system` 后续开发的主执行规范。

它不讨论愿景，不做营销表述，只定义三件事：

- 什么工作才算对系统分数提升有效
- 什么技术改造必须按顺序执行
- 什么行为一律禁止，避免系统继续停留在“看起来完整，实际上证据不足”的阶段

本文档目标是把系统从当前约 `6.3/10` 推进到 `8.0+/10`，并具备对标行业顶级研究决策平台的基础条件。

---

## 2. 目标定义

### 2.1 总目标

在不牺牲当前治理纪律的前提下，把系统升级为：

- 可审计
- 可复现
- 可追责
- 可回滚
- 可持续证明策略质量

### 2.2 8分以上的真实定义

达到 `8.0+/10` 不等于“功能很多”，而等于以下条件同时成立：

- 主事实源唯一，结果不会串档
- 关键运行链每天可连续执行
- 策略质量有固定证据，不靠单次漂亮结果
- 页面展示不能反向篡改制度事实
- 发布与回滚有完整证据闭环
- 系统能解释为什么有结果，也能解释为什么没有结果

### 2.3 当前定位

当前系统的正式定位冻结为：

`A 股研究决策与主结果治理平台`

在达到 `8.0+/10` 前，不对外宣称为：

- 顶级自动交易系统
- 资金可无限放大的成熟实盘系统
- 已完成行业头部对打验证的平台

---

## 3. 执行原则

### 3.1 证据优先原则

任何开发，如果不能提升下列至少一项，不进入高优先级：

- 结果真实性
- 证据连续性
- 运行稳定性
- 决策可追溯性
- 策略可解释性

### 3.2 主事实源原则

所有结果对象都必须存在唯一主事实源。

禁止：

- 多个 latest 文件共同决定一个主结论
- 页面层拼接多个冲突口径生成“看起来合理”的结论
- 用展示层字段替代制度层字段

### 3.3 不可变产物原则

所有研究、回测、候选、审核、执行、观察、终局产物，必须可定位到不可变版本。

`latest` 只允许作为索引入口，不允许作为主事实源。

### 3.4 策略与工程双门禁原则

任何晋升、发布、上线、对外展示增强，必须同时通过：

- 工程门禁
- 策略门禁

工程门禁通过，不代表策略可用。  
策略门禁通过，不代表生产可发。

### 3.5 先治根，再做表现层

以下问题未解决前，禁止把页面优化当成主任务：

- stale artifact
- latest 串档
- 零信号无归因
- 回测与观察链断裂
- release 与 evidence 未绑定

## 4. 顶级化升级主线

后续开发按四阶段执行，不允许跳阶段宣称完成。

### Phase 1: 硬事实底座

目标：消除串档、补齐血缘、统一版本事实。

必须完成：

- 建立 `run_id / result_id / artifact_id` 三层标识
- 把 `latest` 从主读取逻辑中降级为索引
- 建立统一 `artifact_registry`
- 所有主结果支持向上追溯输入工件，向下追溯消费工件
- 所有主结果绑定配置快照、模型快照、数据快照、代码版本

完成标志：

- 任何一个 `/stock` 主结果都能定位唯一证据链
- 任一阶段结果不再依赖多个 latest 文件拼装
- stale artifact 从“被检测”变成“结构上受控”

### Phase 2: 策略证明系统

目标：把“有流程”升级成“有稳定证据”。

必须完成：

- 新增 `strategy_quality_gate`
- 固化 benchmark suite
- 固化 `20 / 60 / 120` 交易日统计口径
- 固化 walk-forward、validation、observation 分层
- 固化零信号、零交易、收益坍塌的失败归因

完成标志：

- 每次研究和晋升都有固定质量报告
- 系统能回答“为什么今天没有信号”
- 对任意候选都能说明是策略问题、市场问题还是规则问题

### Phase 3: 生产级治理与观测

目标：把脚本集合升级为可长期运行的治理平台。

必须完成：

- 统一任务状态模型
- 统一运行监控指标
- 统一异常告警规则
- 发布前后自动复检
- 策略退化可触发人工复核或回滚建议

完成标志：

- 运行问题不是靠人肉发现
- 发布成功不再只代表服务启动成功
- 观察期、回滚期、失败样本都有标准记录

### Phase 4: 顶级系统辨识度

目标：让系统真正具备行业级表达与操作一致性。

必须完成：

- `/stock` 成为唯一业务主结果面
- `/T12` 成为严格只读治理镜像
- 建立主结果档案馆
- 建立版本对比与终局复盘机制

完成标志：

- 用户看到的是单一主结论，不是模块堆叠
- 团队能回答“为什么是它，不是别的”
- 历史主结果可回放、可比较、可复盘

---

## 5. 必做改造清单

### 5.1 事实与数据层

- 建立统一 `run_registry`
- 建立统一 `result_registry`
- 建立统一 `artifact_registry`
- 所有 artifacts 写入 immutable path
- 所有 current 指针只指向 immutable artifact
- 所有 latest 索引必须声明其真实来源

### 5.2 研究与策略层

- 新增 `strategy_quality_gate`
- 增加 benchmark report 与 benchmark diff
- 固定 OOS 观察窗
- 增加 signal drought diagnostics
- 增加 failure attribution taxonomy
- 增加环境分层统计：`bull / bear / range / high-vol`

### 5.3 产品与主结果层

- `/stock` 首页只允许一个 primary conclusion
- 每个主结果卡必须展示：
  - `timestamp`
  - `source artifact`
  - `status`
  - `next action`
- `/T12` 不允许写回行为
- 主结果必须显示晋升原因、阻断原因、观察状态、终局状态

### 5.4 发布与运维层

- 所有 release 绑定 `release_id`
- release 必须绑定 strategy evidence bundle
- 发布后自动执行 verification
- verification 必须覆盖：
  - 路由
  - 主结果 API
  - scope 边界
  - 运行目录保护
  - error log 探测

## 6. 策略门禁标准

### 6.1 策略门禁必备指标

任何候选版本、基线晋升、主结果晋级，必须固定输出：

- 信号覆盖率
- 有交易天占比
- 胜率
- 盈亏比
- 最大回撤
- 超额收益
- 收益回撤比
- 行业集中度
- 容量约束摘要
- 参数敏感性摘要

### 6.2 必须阻断的情况

出现以下任一条件，默认阻断：

- 连续 20 个交易日无信号且无充分解释
- 回测收益为正但观察窗显著失真
- 行业暴露超限
- 参数微调导致排序或收益剧烈漂移
- 关键 artifacts 缺失或指针错位

### 6.3 允许降级观察的情况

以下情况允许进入受控观察，不允许直接晋升：

- 样本量不足但链路完整
- 收益轻微为正但稳定性不足
- 环境覆盖不够厚
- 单次回测成立但跨阶段一致性偏弱

---

## 7. 页面与叙事约束

### 7.1 页面不是主事实源

页面只消费共享事实，不生产制度事实。

禁止：

- 在页面层推导生命周期主字段
- 在页面层合成审核通过结论
- 在页面层用文案掩盖数据缺口

### 7.2 首页约束

首页必须聚焦：

- 当前状态
- 当前主对象
- 当前主阻断
- 当前下一动作
- 当前证据地板进度

禁止首页堆放平级结论卡。

### 7.3 `/apex` 约束

`/apex` 只允许作为 namespace，不允许被表述为第四系统。

---

## 8. 监控与告警标准

必须建立以下监控项：

- 数据同步延迟
- 候选生成延迟
- 主结果生成延迟
- 信号数相对 20 日均值偏离
- 零信号连续天数
- stale artifact 检测
- result pointer 错位
- 发布后 verification 失败
- 观察窗积压
- terminal 失败比例

每个告警都必须有：

- 主因
- 严重级别
- next action
- 责任工件

---

## 9. 严格禁止项

以下行为一律禁止：

- 用单次漂亮结果宣称系统已顶级
- 用页面重构替代底层证据修复
- 继续扩大 `latest` 在主链中的解释权
- 把工程门禁通过表述成策略已成立
- 在没有 benchmark diff 的情况下直接晋升
- 在没有 observation 证据的情况下把结果包装成 production ready
- 让 `/T12` 参与业务主判断
- 用人工记忆替代 registry 和 evidence

## 10. 里程碑与验收

### Milestone A

目标：`7.1/10`

验收条件：

- latest 主依赖拆除
- registry 基本可用
- stale artifact 结构性收敛

### Milestone B

目标：`7.8/10`

验收条件：

- strategy_quality_gate 上线
- benchmark 套件固定
- failure attribution 可用

### Milestone C

目标：`8.2/10`

验收条件：

- 监控和发布复检闭环
- 发布与策略证据绑定
- 运行问题可自动发现

### Milestone D

目标：`8.4/10`

验收条件：

- `/stock`、`/T12`、主站职责稳定
- 主结果档案馆可回放
- 历史主结果可对比、可复盘、可追责

---

## 11. 执行纪律

后续所有开发任务，必须在立项时回答 4 个问题：

1. 它提升的是哪一个维度：事实、证据、稳定性、策略、治理、观测？
2. 它是否减少了系统误判或串档风险？
3. 它的验收证据是什么？
4. 它是否会制造新的页面叙事大于底层事实的问题？

任何任务回答不清这四个问题，默认不进入高优先级。
