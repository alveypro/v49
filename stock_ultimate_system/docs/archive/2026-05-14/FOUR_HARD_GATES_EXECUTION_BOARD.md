# 四道硬门执行总表

## 1. 文档定位

本文档是后续推进的唯一执行总表。

它不讨论愿景，不讨论“像不像头部”，只回答三个问题：

- 现在到底卡在哪四道硬门
- 四道硬门按什么顺序推进
- 每道门什么才算真实过线

执行优先级：

1. `docs/STRICT_CONTINUATION_EXECUTION_STANDARD.md`
2. `docs/TOP_TIER_GAP_MATRIX.md`
3. 本文档

---

## 2. 总体原则

后续推进只围绕以下四道硬门：

1. `现网收敛`
2. `长期稳定`
3. `真实 AI 接入`
4. `长期样本闭环`

禁止项：

- 不平均开四条线
- 不再把“页面优化”误当主任务
- 不在前一门未过线时跳到后一门宣称完成

硬规则：

`必须按 1 -> 2 -> 3 -> 4 的顺序推进。`

## 5. 推进顺序

### 5.1 Gate-1：现网收敛

目标：

- 把“现网能跑”收成“现网配置、服务、路由、定时器、release verification 全收敛”

完成标志：

- formal active nginx 与 repo current 一致
- `/T12` 边界不再依赖热补
- service / timer / route / artifact path 可重复核验
- release verification 有正式产物

未完成前禁止：

- 开始真实 AI provider 接入
- 宣称具备头部级生产托管能力

### 5.2 Gate-2：长期稳定

目标：

- 把“主链已修通”推进成“连续交易日稳定不开口子”

完成标志：

- 连续多日 pointer / lifecycle / guard / api 一致
- verification artifact 连续留存
- release gate 可自动阻断不一致

未完成前禁止：

- 宣称主链已达到头部稳定性

### 5.3 Gate-3：真实 AI 接入

目标：

- 不扩场景，只在当前只读边界内接一个真实 provider

完成标志：

- `/stock` A1 / I1 -> O1 路径有真实 provider adapter
- fail-closed、telemetry、ops read-only 面仍然成立
- `/T12` 不启用，主结果链零耦合

未完成前禁止：

- 扩 `/T12`
- 扩运维建议
- 宣称 AI 已进入成熟生产阶段

### 5.4 Gate-4：长期样本闭环

目标：

- 把“制度设计正确”推进成“长期证据强”

完成标志：

- `20/60/120` 交易日统计成型
- failure attribution 历史成型
- promotion gate 历史成型
- 可回答“这次为什么坏”和“最近为什么老坏”

未完成前禁止：

- 宣称系统已具备头部量化研究闭环证据

## 6. 执行节奏

### 6.1 只允许一个主门

任何一周只允许一个主门占据第一优先级。

当前第一优先级固定为：

`Gate-1：现网收敛`

### 6.2 其余门的工作规则

其余三门只允许做：

- 不阻断当前主门的小修
- 不改变优先级的文档和测试补强

不允许：

- 抢主门优先级
- 借机扩功能

---

## 7. 每道门统一交付格式

每个子项必须同时有三件东西：

1. `文档`
2. `测试 / 校验`
3. `实现 / 部署`

每个工作单元必须固定写成：

- `目标`
- `改动文件`
- `风险`
- `验收命令`
- `放行 / 阻断标准`

---

## 8. 当前唯一执行结论

当前不该平均开工四条线。

当前唯一正确动作是：

`先过 Gate-1：现网收敛。`

在 Gate-1 未过线前：

- 不应把精力继续摊给真实 AI 接入
- 不应把精力继续摊给长期样本闭环
- 不应把产品层打磨误判成主工作
