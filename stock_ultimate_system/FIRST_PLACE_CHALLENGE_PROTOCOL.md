# FIRST PLACE CHALLENGE PROTOCOL

## Status

Version: v1.0  
Status: active operating protocol  
Scope: `/stock` primary result, candidate basket, performance evidence, baseline promotion

## Purpose

This protocol turns the ambition of competing with the best system in the category into a controlled operating standard.

The goal is not to declare leadership by confidence, model count, interface polish, or internal complexity. The goal is to make `/stock` prove that each promoted result is stronger, more explainable, more stable, and more governable than the current baseline and ordinary market benchmarks.

## Non-Negotiable Position

Airivo must compete upward, but promotion is evidence-bound.

- A stronger story is not a stronger system.
- More factors are not stronger signals by default.
- More automation is not higher maturity by default.
- A passed engineering gate does not prove market edge.
- A short winning sample does not prove durable alpha.

The system may challenge the current champion only through repeatable evidence.

## Primary Battlefield

The first battlefield is:

`A-share daily Top N candidate quality`

The system must first prove that its daily candidate output is better than ordinary alternatives before expanding the claim to broader trading automation.

The primary battlefield includes:

- `/stock` canonical primary result
- approved candidate basket snapshot
- closed observation windows
- performance ledger
- benchmark comparison
- failure attribution
- governed baseline promotion

The primary battlefield excludes:

- fully automated live trading
- uncontrolled strategy replacement
- manual after-the-fact result rewriting
- promotion based on screenshots, anecdotes, or single-day wins

## Competitive Baselines

Each challenge round should compare against three baseline classes.

### 1. Market Baselines

- broad market index
- configured benchmark index
- equal-weight candidate basket baseline where available

### 2. Simple Strategy Baselines

- momentum rank
- liquidity rank
- low-volatility filter
- multi-factor score without governance enhancement

### 3. Product Baselines

These are observed manually or through controlled review when direct data is not available:

- common stock screener output
- generic AI stock assistant output
- human-curated watchlist
- brokerage-style short recommendation list

Product baselines must be recorded as review evidence. They must not be silently mixed into automated ledgers unless the source, date, selection rule, and comparison window are recorded.

## Champion Rule

The current approved baseline is the champion.

A challenger may not replace the champion unless it beats the champion under the same rules:

- same stock universe
- same trade calendar
- same observation window
- same transaction-cost assumption where applicable
- same benchmark index
- same Top N size
- same success and drawdown criteria
- same evidence floor
- same release gates

If those conditions are not equal, the challenge is exploratory research, not a promotion candidate.

## Promotion Evidence Floors

The first production-oriented evidence floor remains 20 closed observations.

Evidence windows:

- 20 observations: first promotion review eligibility
- 60 observations: stability review
- 120 observations: durable baseline review

The 20-observation floor allows governed review. It does not prove permanent superiority.

## Minimum Competitive Metrics

Each challenge round must report at least:

- entry total
- Top N size
- basket return
- benchmark return
- excess return
- hit rate or success rate
- maximum drawdown
- worst single-name contribution
- industry concentration
- failure attribution coverage
- data freshness status
- blocked or rejected candidates

For baseline promotion, the first floor must satisfy:

- sample floor met
- success rate floor met
- average excess return floor met
- drawdown floor met
- no blocking release gate failure
- no open high-severity review item
- approved release decision

## Failure Is A Required Artifact

A failed observation is not a reason to hide evidence. It is a required input to the improvement loop.

Each failed observation should be routed to failure attribution with one or more of the following categories:

- data freshness failure
- market regime mismatch
- risk filter miss
- over-concentration
- weak signal quality
- delayed exit signal
- benchmark shock
- unsupported manual override
- unknown cause requiring review

Unknown cause must not be normalized. If unknown causes persist, promotion must stop.

## Candidate Basket Discipline

The candidate basket is the daily competitive surface.

Rules:

- one approved candidate basket per observation cycle
- no after-the-fact item replacement
- no changing weights after the window starts
- no hiding rejected high-score candidates
- no mixing exploratory picks into official basket evidence

If an item is removed by risk rules, the removal reason must be visible in review evidence.

## Baseline Promotion Boundary

Baseline promotion is allowed only after the existing promotion policy is satisfied.

This protocol adds a competitive interpretation:

- `release gates passed` means the system is structurally publishable
- `performance evidence ready` means the result has enough observations for review
- `release decision approved` means a human-governed decision accepts the evidence
- `baseline promoted` means the challenger becomes the new champion

None of the above authorizes live trading by itself.

## Weekly Operating Rhythm

Every week should produce a concise challenge review:

- current champion baseline
- current challenger status
- observation count
- latest excess return
- latest drawdown
- failures and attribution
- blocked promotion reasons
- next controlled change

The weekly review should prefer one high-leverage change over many simultaneous changes.

## Stop Conditions

Promotion must stop if any of the following occurs:

- data freshness cannot be proven
- benchmark comparison is missing
- observation sample is below the required floor
- average excess return is negative at the active floor
- drawdown breaches the floor
- high-severity review items remain open
- failure attribution is repeatedly unknown
- release gates fail
- `/stock` canonical single-track output drifts
- `/T12` governance content leaks into `/stock`

## Operating Conclusion

Airivo competes like a top system only when it behaves like one under pressure:

- it keeps one official result
- it measures against the champion
- it records losses
- it rejects weak promotion
- it improves through evidence

The standard is not whether the system can sound first-class. The standard is whether it can repeatedly earn the right to replace its own champion.
